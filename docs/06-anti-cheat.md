## Protection Layers

### 1. Server-Authoritative Combat

**All combat calculations are on the server.** The client is only a display layer.

- Damage is calculated on BE: `cachedStats.tapDamage` + `critChance` — from Redis cache (at session start), not from client
- Monster HP is stored in Redis session
- Client cannot forge damage or kill a monster
- Completion verifies that all monsters are killed
- Enemy attacks are generated by the server every 200ms (combat loop)
- Elemental damage (physical/fire/lightning/cold/pure) is calculated on the server

```typescript
// combat.service.ts - processTap()
const stats = session.cachedStats;         // cached at start, not from client
const isCrit = Math.random() < stats.critChance;  // server-side RNG
let damage = stats.tapDamage;
if (isCrit) damage = Math.floor(damage * stats.critMultiplier);
// + elemental damage calculation
monster.currentHp -= totalDamage;          // server-side HP
```

### 2. Universal Socket Rate Limiter

**A single rate limiter for ALL WebSocket messages.** Replaced the old per-tap `checkTapRate`.

| Parameter | Value | Constant |
|-----------|-------|----------|
| Window | 3 seconds | `B.ANTICHEAT_WINDOW_MS` |
| Limit | 30 messages / window | `B.ANTICHEAT_MSG_LIMIT` |
| Punishment | Instant 24h ban | `B.ANTICHEAT_BAN_DURATION_MS` |

```typescript
// combat.service.ts
async checkMessageRate(telegramId: string): Promise<boolean> {
  const windowIndex = Math.floor(Date.now() / B.ANTICHEAT_WINDOW_MS);
  const key = `anticheat:msgs:${telegramId}:${windowIndex}`;  // TTL 10s
  const count = await this.redis.incr(key);
  if (count === 1) await this.redis.expire(key, 10);
  if (count <= B.ANTICHEAT_MSG_LIMIT) return false; // OK
  await this.banPlayer(telegramId, 'rate_limit_exceeded');
  return true; // BANNED
}
```

Called in **every** WebSocket handler via `rateLimitCheck()`:

```
handleStartLocation  handleStartMap     handleStartDojo
handleTap            handleCompleteDojo handleComplete
handleFlee           handleEntranceDone handleReconnect
handleUsePotion      handleCastSkill
```

Additionally: HTTP throttling via NestJS ThrottlerModule:

| Level | Setting | Where |
|-------|---------|-------|
| Global | 100 req / 60 sec | `app.module.ts` ThrottlerModule |
| Auth | 5 req / 60 sec | `auth.controller.ts` @Throttle |

### 3. Ban System

#### Trigger

The only automatic trigger: `rate_limit_exceeded` — exceeding 30 socket messages in a 3-second window.

#### Dual Storage (Redis + PostgreSQL)

Ban is stored in **two places** simultaneously:

| Storage | Key / Field | TTL | Purpose |
|---------|-------------|-----|---------|
| **Redis** | `anticheat:ban:{telegramId}` | 24h (auto-expire) | Fast path — check in ~1ms |
| **PostgreSQL** | `Player.bannedUntil` + `Player.banReason` | none (persistent) | Durability — survives Redis flush |

```typescript
// combat.service.ts - banPlayer()
private async banPlayer(telegramId: string, reason: string): Promise<void> {
  const expiresAt = Date.now() + B.ANTICHEAT_BAN_DURATION_MS;

  // 1. Redis — fast check (auto-expires)
  await this.redis.setJson(`anticheat:ban:${telegramId}`, {
    reason, bannedAt: Date.now(), expiresAt
  }, Math.ceil(B.ANTICHEAT_BAN_DURATION_MS / 1000));

  // 2. PostgreSQL — persistent
  await this.playerRepo.update(telegramId, {
    bannedUntil: new Date(expiresAt),
    banReason: reason,
  });
}
```

#### Ban Check with fallback

```typescript
// combat.service.ts - isPlayerBanned()
async isPlayerBanned(telegramId): Promise<{ banned: boolean; expiresAt?: number }> {
  // 1. Fast path: Redis
  const banData = await this.redis.getJson(`anticheat:ban:${telegramId}`);
  if (banData?.expiresAt > Date.now()) return { banned: true, expiresAt: banData.expiresAt };

  // 2. Fallback: DB (if Redis was flushed)
  const player = await this.playerRepo.findOne({ where: { telegramId } });
  if (player?.bannedUntil?.getTime() > Date.now()) {
    // Re-cache in Redis for subsequent checks
    await this.redis.setJson(`anticheat:ban:${telegramId}`, { ... }, remainingTtl);
    return { banned: true, expiresAt: player.bannedUntil.getTime() };
  }

  return { banned: false };
}
```

#### 5 Ban Enforcement Points

| # | Where | File | What happens |
|---|-------|------|--------------|
| 1 | **Login** | `auth.service.ts` → `authenticateTelegram()` | Returns `banned: true` + `bannedUntil` in response |
| 2 | **State loading** | `player.service.ts` → `getPlayerState()` | Returns only ban info, without heavy data |
| 3 | **WS connection** | `combat.gateway.ts` → `handleConnection()` | `emit('combat:banned')` + `disconnect()` |
| 4 | **Every WS message** | `combat.gateway.ts` → `rateLimitCheck()` | Session cleanup + `emit('combat:banned')` |
| 5 | **In combat (event)** | `combat.gateway.ts` | `combat:banned` event to client |

#### Client-side handling

**On game startup** (`state.ts` + `main.ts`):

```typescript
// state.ts - BannedError
export class BannedError extends Error {
  bannedUntil: number;
  banReason: string;
}

// state.ts - load()
const authResult = await api.auth.login(initData);
if (authResult.player.banned) {
  throw new BannedError(authResult.player.bannedUntil, authResult.player.banReason);
}

// main.ts - bootstrap
catch (err) {
  if (err instanceof BannedError) {
    showBanScreen(err.bannedUntil, err.banReason); // fullscreen ban
    return; // game does not start
  }
}
```

**During combat** (`combat-scene.ts`):

```
combat:banned event → ban overlay (semi-transparent)
  → shows ban expiration time
  → "Return to Hideout" button
```

**During dojo** (`dojo-scene.ts`):

```
combat:banned event → ban overlay
  → shows remaining hours
  → "Return to Hideout" button
```

#### Manual unban

Both storages must be cleared:

```sql
-- PostgreSQL
UPDATE player SET "bannedUntil" = NULL, "banReason" = NULL WHERE "telegramId" = '123456';
```

```bash
# Redis
docker exec 33metro-redis-1 redis-cli DEL anticheat:ban:123456
```

If only DB is cleared — Redis cache will still enforce the ban (until TTL expires).
If only Redis is cleared — DB fallback will recreate the Redis entry.

### 4. Server-Authoritative Dojo

Dojo (training dummy) is entirely server-side via WebSocket.

**Socket Events:**

| Direction | Event | Payload |
|-----------|-------|---------|
| C→S | `combat:start-dojo` | `{}` |
| S→C | `combat:dojo-started` | `{ sessionId, dojoEndsAt }` |
| C→S | `combat:tap` | `{ sessionId }` |
| S→C | `combat:tap-result` | `{ damage, isCrit, totalDamage, totalTaps, dojoEnded, timeLeftMs }` |
| C→S | `combat:complete-dojo` | `{ sessionId }` |
| S→C | `combat:dojo-completed` | `{ totalDamage, totalTaps, dps, bestDamage, isNewBest }` |

**Protection:**
- Server-side timer: `dojoEndsAt = now + DOJO_COUNTDOWN_MS + DOJO_ROUND_MS`
- Auto-complete on server: `setTimeout(completeDojo, COUNTDOWN + ROUND + 500ms)` — cannot play longer
- Damage = server-side `tapDamage × critMultiplier` from cached stats
- Anti-double-fire guard: `if (this._phase === "results") return`
- Rate limiter works in dojo too (same 30 msgs / 3 sec)
- Client-side 100ms rate limit — for UX only (debounce), not for security

### 5. Telegram Signature Verification

HMAC-SHA256 verification of initData — impossible to forge authentication without BOT_TOKEN.

```
secretKey = HMAC-SHA256('WebAppData', BOT_TOKEN)
checkHash = HMAC-SHA256(secretKey, dataCheckString)
// + check that auth_date is no older than 24 hours
```

### 6. JWT Security

- Separate access / refresh tokens
- Access: 1 hour (short-lived)
- Refresh: 30 days (Redis stored, verifiable)
- Token type validation: `payload.type === 'access'` (refresh token won't pass as access)
- WebSocket: JWT is verified on connection (`client.handshake.auth.token`)
- Invalid token → disconnect
- Mapping: socketId <-> telegramId (one socket per user)

### 8. Session Ownership

```typescript

  throw new NotFoundException(); // cannot tap in someone else's session
}
```

Sessions in Redis with TTL 30 min (combat) / 60 sec (dojo) — cannot be reused.

2. All IDs valid (integers, in range)
3. BFS connectivity — all nodes are connected to the start node
4. Budget: outer <= level-1, classSkills <= 6

### 10. Player Ownership on all requests

All DB queries include `playerTelegramId = telegramId`:

```typescript
charRepo.findOne({ where: { id: characterId, playerTelegramId: telegramId } })
```

Impossible to access another player's data.

### 11. Audit Trail

`CombatSession` entity is saved to PostgreSQL:
- totalMonsters, monstersKilled, totalGoldEarned, totalXpEarned
- totalTaps, timestamp, status (active/completed/abandoned/flagged)

Enables post-game anomaly analysis.

### 12. Anti-AFK in combat

`MAX_PENDING_ATTACKS = 10` — maximum of 10 monster attacks per tick.
Prevents accumulating infinite timeBank.

### 13. Disconnect Grace Period

30-second grace period on disconnect.
After 30s — automatic flee (combat lost, no rewards).
On reconnect: `lastEnemyAttackTime = now` — "forgiveness" for missed attacks.
In dojo: disconnect → completeDojo() (score is saved).
